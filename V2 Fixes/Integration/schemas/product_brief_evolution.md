# Product Brief Evolution Guide

**Purpose:** The Product Brief is a single evolving document that grows richer as it passes through the pipeline. This guide documents which fields are added at each stage and which system is responsible.

---

## Stage Progression

```
Stage 0 (Seed)         → 5 core fields
Stage 1 (Post-Research) → + 8 research-derived fields
Stage 2 (Post-Angle)    → + 6 angle/compliance fields
Stage 3 (Post-Offer)    → + 9 offer architecture fields
```

---

## Field Inventory

### Stage 0: Seed Input (Operator fills)

| Field | Type | Required | Source |
|-------|------|----------|--------|
| `product_name` | string | Yes | Operator |
| `description` | string | Yes | Operator |
| `price` | string | No | Operator (or TBD) |
| `competitor_urls` | string[] | No | Operator |
| `product_customizable` | boolean | Yes | Operator |

### Stage 1: Post-Research (System generates from Foundational Docs)

| Field | Type | Source System | Specific Output |
|-------|------|--------------|-----------------|
| `category_niche` | string | Step 01 | STEP1_CONTENT header |
| `market_maturity_stage` | enum | Step 01 | Market stage assessment section |
| `primary_segment` | object | Step 06 | STEP6_CONTENT top-ranked segment |
| `bottleneck` | string | Step 06 | STEP6_CONTENT bottleneck analysis |
| `positioning_gaps` | string[] | Step 01 | STEP1_CONTENT whitespace section |

| Field | Type | Source System | Specific Output |

| `purchase_emotion` | string | Agent 3 | Angle's dominant emotional valence |

## Usage Notes

1. **Additive only:** Each stage adds fields. No fields are removed or renamed between stages.
2. **Backward compatible:** A Stage 3 brief contains all Stage 0, 1, and 2 fields.
3. **Single file:** Maintain one `product_brief.json` (or `.yaml`) file and update it at each stage.
4. **Validation:** Use the JSON Schema files in `integration/schemas/` to validate at each stage.
5. **Operator fields:** Some fields at Stage 2 and 3 require operator judgment (e.g., `buyer_behavior_archetype`). These are marked as "Inferred" in the source column — the operator confirms or overrides the system's suggestion.
